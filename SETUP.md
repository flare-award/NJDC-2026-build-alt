Без настройки Supabase (см. ниже) вход всё равно работает — изменения будут сохраняться локально в вашем браузере (localStorage). Как только вы подключите Supabase, изменения станут видны **всем посетителям сайта в реальном времени**.

## 2. Подключение Supabase (чтобы изменения видели все посетители)

1. Зарегистрируйтесь на [supabase.com](https://supabase.com) и создайте новый проект.
2. Откройте **SQL Editor** в панели проекта и выполните целиком файл `supabase-schema.sql` из корня этого проекта — он создаст все нужные таблицы (`teams`, `players`, `matches`, `votes`, `faq`, `settings`), настроит политики безопасности (RLS) и заполнит начальными данными.
3. Откройте **Authentication → Users** и создайте пользователя вручную:
   - Email: `maronn@njdc.local`
   - Пароль: `9991`
   
   Это нужно, чтобы кнопки сохранения в админ-панели реально писали данные в базу (запись разрешена только авторизованным пользователям — это защищает базу от посторонних, даже если кто-то найдёт публичный ключ анон-доступа).
4. Откройте **Storage** → создайте новый bucket с именем `logos`, отметьте его как **Public**. Сюда будут загружаться логотипы команд из админ-панели.
5. Откройте **Project Settings → API** и скопируйте `Project URL` и `anon public` ключ.
6. В корне проекта создайте файл `.env` (по образцу `.env.example`) и вставьте туда эти значения:
   ```
   VITE_SUPABASE_URL=https://xxxxx.supabase.co
   VITE_SUPABASE_ANON_KEY=eyJhbGciOi...
   ```
7. Пересоберите проект (`npm run build`) — переменные окружения "запекаются" в сборку на этапе билда, поэтому после любого изменения `.env` нужно пересобирать сайт перед публикацией на GitHub Pages.

Если `.env` не заполнен — сайт продолжает работать в автономном режиме (демо-данные + localStorage), ничего не ломается.

## 3. Регламент турнира (PDF)

В футере сайта есть кнопка «Открыть регламент (PDF)», которая ведёт на файл:

```
public/NJDC 2026 regulations.pdf
```

Сейчас там лежит **временная заглушка** (на английском, так как стандартные PDF-шрифты не поддерживают кириллицу без встраивания шрифта). Замените этот файл своим финальным регламентом на русском языке — **сохраните точно такое же имя файла**, и ссылка в футере продолжит работать без изменений кода.

## 4. Логотипы команд

Логотипы лежат в `public/logos/team-a.png` … `team-f.png` (сгенерированы автоматически, в виде эмблем на тёмном фоне под стиль сайта). Заменить их можно двумя способами:

- Через админ-панель → вкладка «Команды» → загрузить файл (работает только при подключённом Supabase Storage, см. выше) или просто вставить прямую ссылку на изображение (URL) в поле «Логотип».
- Вручную — заменить файлы в папке `public/logos/` перед сборкой.

Рекомендуется использовать PNG с прозрачным или тёмным (#0D0D0D) фоном.

## 5. Публикация на GitHub Pages

1. `npm run build` — соберёт сайт в папку `dist/`.
2. Залейте содержимое `dist/` в ветку `gh-pages` (или настройте GitHub Actions деплой из `dist/`).
3. Роутинг сайта использует `HashRouter` (ссылки вида `#/matches`), поэтому GitHub Pages будет корректно открывать любые внутренние страницы без дополнительной настройки 404.html.

## 6. Как редактировать турнир изо дня в день

Всё редактируется через админ-панель (`/#/nb-admin-9991`):

- **Матчи** — создание матчей по этапам, ввод счёта, статуса (скоро/LIVE/завершён), ссылки на CYBERSHOKE. Очки в таблице считаются автоматически по формату (Bo1/Bo2/Bo3).
- **Команды** — название, состав (текстом), логотип, рейтинг силы.
- **Игроки** — K/D, Faceit Elo, рейтинг для таблицы лидеров, заметки.
- **FAQ** — вопросы/ответы внизу сайта.
- **Настройки** — заголовки главной страницы, объявление, текст приза, путь к PDF регламента.

Голоса зрителей (прогнозы/коэффициенты) на страницах матчей считаются автоматически из таблицы `votes` — вручную ничего считать не нужно, при желании голоса по конкретному матчу можно сбросить кнопкой в вкладке «Матчи».
